ตอบ❓คำถามเพื่อความชัดเจน:
1. User & Role Management
Multi-role: User 1 คนสามารถมีหลาย Role ได้ แต่มีข้อจำกัดว่า Requester ไม่สามารถเป็น Purchasing และ Approver - ถูกต้องใช่ไหมครับ?
ตอบ  ถูกต้อง
Role hierarchy: Approver มีหลาย level ตามลำดับชั้น - จำนวน level สูงสุดเท่าไหร่ครับ? หรือไม่จำกัด?
ตอบ  ไม่เกิน 3
Department binding: User บางคนมีหลายบริษัท/หลายฝ่ายงาน - binding นี้ผูกกับ Role ด้วยหรือไม่? (เช่น User A เป็น Requester ที่บริษัท X แต่เป็น Approver ที่บริษัท Y ได้ไหม?)
ตอบ	จากการยกตัวอย่าง เช่น User A เป็น Requester ที่บริษัท X แต่เป็น Approver ที่บริษัท Y ได้ไหม? ไม่ได้อยู่แล้วจะผิดกฤข้อที่ 1 Requester ไม่สามารถเป็น Purchasing และ Approver แต่ถ้า User A เป็น Requester ที่บริษัท X แต่เป็น Requester ที่บริษัท Y ได้
2.Category & Subcategory
Serial Number: ในเอกสารระบุว่า Subcategory มี field IsUseSerial (Y/N) - Serial Number นี้ใช้ทำอะไรครับ?
ตอบ	ถ้า Subcategory ถูก ตั้งค่าให้ สมมุติ IsUseSerialแปลว่า การสร้าง สร้างใบขอราคา input Serial Number ต้องบังคับกรอก
Duration: Subcategory มี field Duration สำหรับคำนวณวันที่ต้องการใบเสนอราคา - Duration นี้เป็นจำนวนวันใช่ไหมครับ?
ตอบ	ใช่
Document Requirements: Subcategory กำหนด document ที่ต้อง upload (ชื่อ, IsRequired) - มี limit จำนวน document ต่อ Subcategory ไหมครับ?
ตอบ	ไม่มี และ ไม่เกิน 30 mb ต่อไฟล์
3.RFQ Number Format
เอกสารมี 2 format:
-Com-yy-mm-xxxx (ตัวอย่างทั่วไป)
-ShortNameEn-yy-mm-xxxx (format จริง เช่น ปป-01-0001)
ตอบ	ใช้  ShortNameEn-yy-mm-xxxx
ShortNameEn มาจากไหนครับ? (Company? Department? Category?)
ตอบ	ชื่อย่อCompany +  ปี-เดือน-xxxx (runging code เริ่มต้น 0001เพิ่มไปเรื่อยๆ)
4.Supplier Management
Multiple Contacts: Supplier มีได้หลาย contact และส่ง email ไปทั้งหมด - มี primary contact ไหมครับ?
ตอบ	มี
Supplier Categories: Supplier ผูกกับ Category/Subcategory - 1 Supplier สามารถอยู่หลาย Category ได้ไหมครับ?
ตอบ	ได้
5.Exchange Rate
Update Frequency: อัปเดตเดือนละครั้ง - เก็บ history ของ exchange rate ไหมครับ? (สำหรับดูย้อนหลัง)  
ตอบ	ใช่ เก็บ history
Currency Calculation: ถ้า Supplier เสนอราคาเป็นสกุลเงินต่างจาก Company - ใช้ rate ณ วันที่ใดในการคำนวณครับ? (วันที่เสนอราคา? วันที่สร้าง RFQ?)
ตอบ	Lock rate ไว้ ณ วันเวลาที่ปิดการเสนอหรือหมดเวลาเสนอราคา
6.Workflow Status
Soft Delete: ระบุว่า delete user เป็น soft delete - RFQ, Quotation และ entity อื่นๆ ใช้ soft delete ด้วยไหมครับ?
ตอบ	ใช้กับ user เท่านั้น
Status Tracking: ต้องการเก็บ history ของการเปลี่ยน status (audit log) ไหมครับ?
ตอบ	เก็บ history ทุก status change
7.Notification System
Notification Types: มีทั้ง in-app และ email - ต้องเก็บ read/unread status ไหมครับ?
ตอบ	ต้องเก็บ
Reminder Logic: ระบุว่าถ้าไม่มี action ใน 2 วัน จะแจ้งเตือน - logic นี้ต้องเก็บ last action timestamp ไหมครับ?
ตอบ	เก็บ last_action_at และ reminder timestamps
8.File Management
File Storage: เอกสารที่ upload (PDF, Excel, etc.) - เก็บ path อย่างเดียว หรือต้องเก็บ metadata อื่นๆ ด้วยครับ? (size, mime type, uploaded by, etc.)
ตอบ	uploaded by
File Versioning: ถ้า Supplier แก้ไขใบเสนอราคา - เก็บ version เก่าไว้ไหมครับ?
ตอบ	ไม่เก็บ
9.Q&A System
Thread Structure: ถาม-ตอบระหว่าง Supplier กับ Purchasing - เป็น thread แบบ linear หรือมี reply to specific message ครับ?
ตอบ	Linear per Supplier (แยก thread ตาม Supplier)
10.Performance Metrics
On Time/Delay: ต้องเก็บ metrics อะไรบ้างครับ? (response time, processing time, etc.)
ตอบ	เก็บ timestamp ทุกจุดสำคัญ, ใช้ materialized view สำหรับ dashboard
Dashboard Data: Dashboard แสดงข้อมูล realtime - ต้องมี materialized view หรือ caching strategy ไหมครับ?
ตอบ	มี
